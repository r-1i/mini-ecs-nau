# ADR-0001: Хранение компонентов, sparse-set пулы

## Как было

`Scene` хранит компоненты так:

```cpp
std::unordered_map<
    Entity, std::unordered_map<std::type_index, std::unique_ptr<Component>>>
    entities_;
```

На каждую сущность своя хеш-мапа компонентов, каждый компонент отдельная аллокация в куче через `unique_ptr`. Просто, универсально для любого типа компонента, и с этого было правильно начинать, чтобы сначала получить рабочее и протестированное ядро (issues 1-8).

Проблема в том, как это лежит в памяти. `getEntitiesWith<T>()`, который дергает каждая система каждый кадр, на каждую сущность делает хеширование `type_index` и разыменование `unique_ptr`, указывающего на случайный адрес в куче. Замерено профилировщиком (issue 10) на 100 000 сущностей, 200 вызовов `update()`, Release-сборка:

- `Scene::update`: 31.5 мс в среднем
- Заполнение сцены: 85 мс
- Память: ~16.4 МБ

## Как будет

Вместо одной мапы на всех отдельный packed-контейнер (`ComponentPool<T>`) на каждый тип компонента, спрятанный за общий интерфейс `IComponentPool`, чтобы `Scene` могла хранить пулы разных типов в одной коллекции.

Внутри `ComponentPool<T>`:
- `std::vector<T> m_dense`: сами данные, подряд в памяти;
- `std::vector<Entity> m_denseToEntity`: какой сущности принадлежит `m_dense[i]`;
- `std::vector<size_t> m_sparse`: индексируется напрямую по `Entity::id()`, без хеширования, даёт индекс в `m_dense` за одно обращение к массиву.

Удаление swap-and-pop, чтобы `m_dense` никогда не разрежался дырами.

Публичный API `Scene` не меняется: `addComponent<T>`, `getComponent<T>`, `hasComponent<T>`, `getEntitiesWith<Ts...>`, `update`, `saveToFile`, `loadFromFile` те же сигнатуры. Меняется только то, что у них внутри. Поэтому уже написанные тесты (`scene_tests`, `movement_tests`, `serialization_tests`, `memory_tests`) становятся бесплатной регрессионной проверкой миграции.

## Почему: память, производительность, кэш процессора

**Кэш процессора.** Компонент это не только сами данные, это ещё и то, где физически лежат байты. Сейчас `TransformComponent` каждой сущности размещён отдельным `new` в случайном месте кучи. Когда система идёт по 1000 сущностям подряд, процессор 1000 раз прыгает по непредсказуемым адресам, почти на каждый переход cache miss (нужные данные не в кэше, приходится ждать похода в оперативную память). В новой схеме все `TransformComponent` лежат в одном `std::vector` подряд: итерация это последовательное чтение памяти, кэш-линии подгружаются заранее (аппаратный prefetch), промахов на порядки меньше.

**Производительность.** Сейчас каждое обращение (`getComponent`, `hasComponent`, и внутри них каждая проверка в `getEntitiesWith`) это хеш `type_index` плюс хеш `Entity`, то есть минимум два вычисления хеш-функции и поход по бакетам хеш-таблицы. С `m_sparse`, индексируемым напрямую по `id()`, это превращается в обращение к массиву по индексу: O(1) без единого вычисления хеша.

**Память.** Сейчас каждый `addComponent` это отдельный вызов аллокатора (`make_unique`), плюс накладные расходы самой хеш-таблицы (служебные узлы, бакеты) на каждую пару entity-компонент. `std::vector::emplace_back` в пуле аллоцирует блоками (амортизированный рост), а не по одному объекту: меньше отдельных выделений, меньше фрагментации кучи, меньше накладных расходов на объект.

## Результат

Тот же бенчмарк (100 000 сущностей, 200 вызовов `update()`, Release), после миграции на `ComponentPool<T>`:

| | было | стало |
|---|---|---|
| `Scene::update` avg | 31.5 мс | 6.6 мс |
| Заполнение сцены | 85 мс | 13.2 мс |
| Память | 16.4 МБ | 8.44 МБ |

Публичный API `Scene` не менялся, все ранее написанные тесты (`scene_tests`, `movement_tests`, `serialization_tests`, `memory_tests`) прошли без единой правки в самих тестах.

## Отказ от RTTI

Coding style Nau Engine запрещает RTTI (`typeid`, `dynamic_cast`, `std::type_index`). Изначальная версия `m_pools` использовала `std::type_index(typeid(T))` как ключ, а `saveToFile` определял, реализует ли компонент `ISavable`, через `dynamic_cast<ISavable*>`. Оба места заменены, публичный API и производительность не изменились:

- Ключ `m_pools` теперь свой `TypeId` (`size_t`): каждый тип получает уникальное значение через статический счётчик при первом вызове `getTypeId<T>()` для этого типа. Обычное значение `size_t` в хеш-таблице вместо `type_index`.
- `dynamic_cast<ISavable*>` заменён на `IComponentPool::getSavable(Entity)`. Каждый `ComponentPool<T>` реализует его через `if constexpr (std::is_base_of_v<ISavable, T>)`: реализует ли `T` `ISavable`, решается на этапе компиляции, а не рантайма через RTTI.
