## Технические рамки

- Язык: C++20.
- Сборка: CMake (версия 3.16 и выше).
- Графические API и сторонние движки не используются.
- Сторонняя библиотека: [nlohmann/json](https://github.com/nlohmann/json) (v3.11.3), подключается через CMake FetchContent.
- RTTI (`typeid`, `dynamic_cast`, `std::type_index`) в проекте намеренно не используется, вместо этого типы компонентов идентифицируются через собственный механизм `TypeId` (см. `type_id.h`). Подробное обоснование — в `docs/adr/0001-component-pool-storage.md`.

## Сборка и запуск

```bash
git clone https://github.com/r-1i/mini-ecs-nau.git
cd mini-ecs-nau

cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release

# Linux / macOS:
./build/mini-ecs-nau

# Windows (Visual Studio генератор, multi-config):
./build/Release/mini-ecs-nau.exe
```

Требуется компилятор с поддержкой C++20 и CMake версии 3.16 или выше. Зависимость nlohmann/json подтягивается автоматически при конфигурации через FetchContent, поэтому для первой сборки нужно интернет-соединение.

Если в системе доступен [sccache](https://github.com/mozilla/sccache), CMake автоматически задействует его как `CMAKE_CXX_COMPILER_LAUNCHER` для ускорения повторных сборок.

## Цели сборки

Проект собирает одну основную программу и несколько дополнительных целей, использующих те же исходники ядра (`ECS_SOURCES`/`ECS_HEADERS`):

| Цель | Назначение |
|---|---|
| `mini-ecs-nau` | Основная программа, точка входа `src/main.cpp` — демонстрация работы ECS. |
| `scene_tests` | Тесты `Scene` (создание/удаление сущностей, работа с компонентами). |
| `movement_tests` | Тесты `MovementSystem`. |
| `memory_tests` | Тесты, связанные с корректностью работы с памятью. |
| `serialization_tests` | Тесты сохранения/загрузки сцены (`saveToFile`/`loadFromFile`). |
| `scene_benchmark` | Не тест на корректность, а замер производительности (`docs/adr/0001-component-pool-storage.md`, issue 10). Собирать и запускать только в конфигурации Release — цифры в Debug непоказательны. |

Запуск конкретной цели после сборки, например:

```bash
./build/scene_tests
./build/scene_benchmark
```

Все цели используют precompiled header `<nlohmann/json.hpp>`, собранный один раз для `mini-ecs-nau` и переиспользуемый остальными (`REUSE_FROM`) — это заметно ускоряет повторную сборку тестов и бенчмарка.

## Архитектура

Проект построен вокруг четырёх базовых сущностей ECS:

- **Entity** (`entity.h`): лёгкая обёртка над `uint32_t`. Конструктор помечен `explicit`, чтобы исключить случайное неявное преобразование произвольного целого числа в идентификатор сущности. Для использования `Entity` в качестве ключа `unordered_map`/`unordered_set` реализована специализация `std::hash`.

- **Component** (`component.h`): пустой базовый класс компонента с виртуальным деструктором. 

- **ComponentPool\<T\>** (`component_pool.h`) - типизированное хранилище всех компонентов одного типа `T`, построенное по схеме sparse set. Подробности ниже, в разделе [Хранение компонентов и производительность](#хранение-компонентов-и-производительность).

- **Scene** (`scene.h` / `scene.cpp`) центральный класс, управляющий сущностями, пулами компонентов и системами. Предоставляет:
  - `createEntity()` / `destroyEntity(Entity)`;
  - `addComponent<T>(Entity, Args...)`, `getComponent<T>(Entity)`, `hasComponent<T>(Entity)`;
  - `getEntitiesWith<Ts...>()` выборку сущностей, обладающих одновременно всеми перечисленными типами компонентов, с прямыми ссылками на сами компоненты;
  - `registerSystem(std::unique_ptr<System>)` и `update(dt)` регистрацию и прогон систем;
  - `saveToFile(path)` / `loadFromFile(path)` сериализацию сцены в JSON и обратно;
  - `memoryBytes()` приблизительную оценку памяти, занятой всеми пулами.

- **System** (`system.h`) базовый класс логики, работающей над `Scene`. Ссылка на сцену передаётся один раз через конструктор, а не при каждом вызове `execute`, и вместо синглтона `Scene` внедряется явно, это упрощает тестирование системы в изоляции.

Поскольку конкретные типы компонентов не известны `Scene` заранее, разные `ComponentPool<T>` хранятся в единой карте `m_pools` через типостёртый интерфейс `IComponentPool` (`i_component_pool.h`). Ключом карты выступает `TypeId` — уникальный числовой идентификатор типа, выдаваемый функцией `getTypeId<T>()` (`type_id.h`) на основе статического счётчика внутри шаблонной функции. Это позволяет полностью отказаться от RTTI, оставаясь при этом типобезопасным на уровне компиляции.

## Компоненты

### TransformComponent

Позиция, вращение и масштаб сущности. Каждый вектор из трёх `float` (`Vec3`). Масштаб по умолчанию инициализируется единицей по всем осям.

### HealthComponent

Текущее и максимальное здоровье. Предоставляет методы `takeDamage(amount)` и `heal(amount)`, корректно ограничивающие результат снизу нулём и сверху максимальным значением здоровья.

Оба компонента реализуют интерфейс `ISavable` (см. ниже) и потому участвуют в сохранении/загрузке сцены.

## Системы

### MovementSystem

Проходит по всем сущностям, обладающим `TransformComponent`, и на каждом кадре сдвигает их позицию на `velocity * dt`. Демонстрация работы системы (создание сущностей, несколько кадров `update()`, вывод позиций до и после) реализована в `main.cpp`, корректность покрыта `movement_tests`.

## Сохранение и загрузка сцены

Реализован вариант А из части 2 задания: Сохранение и загрузка состояния `Scene`.

**Формат:** JSON, через библиотеку `nlohmann/json`.

Причины выбора:

- Человекочитаемый формат, легко отлаживать и проверять содержимое файла сцены вручную, без дополнительных инструментов.
- `nlohmann/json`, естественный синтаксис работы с JSON-объектами, подключается одной строкой через CMake FetchContent.
- Компактность не являлась требованием задания, поэтому текстовый формат предпочтён бинарному ради простоты диагностики ошибок сериализации.

**Механизм.** Каждый сохраняемый компонент реализует `ISavable::toJson()`, возвращающий JSON-объект с обязательным полем `"type"` — строковым именем класса. При загрузке `Scene::loadFromFile`:

1. Строка `"type"` ищется в `ComponentRegistry` (`component_registry.h/.cpp`) — статической карте, сопоставляющей имя типа с тройкой `{fromJson, makePool, typeId}`.
2. Фабрика `fromJson` восстанавливает типостёртый `unique_ptr<Component>` из JSON.
3. Если пул для этого `typeId` ещё не создан (например, тип впервые встретился при чтении файла), он создаётся фабрикой `makePool`.
4. Компонент передаётся в пул через `IComponentPool::insertFromBase`, где приводится к настоящему типу `T` и перемещается в типизированное хранилище.

Загрузка производится во временный объект `Scene`; текущее состояние сцены заменяется им только при полном успехе разбора файла. Если файл отсутствует, повреждён (перехватывается `nlohmann::json::exception`) или содержит неизвестный тип компонента, `loadFromFile` возвращает `false`, а исходная сцена остаётся без изменений. Корректность сериализации/десериализации покрыта `serialization_tests`.

## Выбор указателей

В проекте `std::unique_ptr` используется на всех участках, где требуется владение объектом:

- `Scene::m_pools` — карта `unordered_map<TypeId, std::unique_ptr<IComponentPool>>`. Сцена — единственный владелец каждого пула, разделяемое владение не требуется.
- `Scene::m_systems` — `vector<std::unique_ptr<System>>`, по той же причине.
- Промежуточные типостёртые объекты при загрузке из JSON (`ComponentRegistry::fromJson`, `IComponentPool::insertFromBase`), компонент существует как `unique_ptr<Component>` ровно до момента перемещения его содержимого в типизированный пул.

`shared_ptr` в проекте не используется: ни у одного объекта нет нескольких одновременных владельцев с неопределённым временем жизни каждого. Время жизни компонентов, пулов и систем однозначно определяется временем жизни `Scene`. Использование `shared_ptr` добавило бы накладные расходы на atomic-счётчик ссылок без реальной необходимости в разделяемом владении.

Сами компоненты внутри `ComponentPool<T>` при этом не хранятся через указатели вовсе, см. раздел далее. Корректность управления памятью (отсутствие утечек) проверяется `memory_tests`.

## Хранение компонентов и производительность

`ComponentPool<T>` реализован по схеме sparse set (полная история решения в `docs/adr/0001-component-pool-storage.md`):

- `m_dense` — `std::vector<T>`, компоненты хранятся непрерывным блоком по значению, без индивидуального выделения памяти под каждый.
- `m_denseToEntity` — соответствие индекса в `m_dense` сущности-владельцу.
- `m_sparse[entity.id()]` — индекс компонента данной сущности в `m_dense`, без хеширования.

Итерация по всем компонентам одного типа (`data()`, `getEntitiesWith`) идёт по непрерывной памяти, это кэш-дружественно и не требует разыменования указателей на каждый элемент, в отличие от изначального варианта (по одному `unordered_map<type_index, unique_ptr<Component>>` на сущность, с отдельной аллокацией на каждый компонент и отдельным хешированием на каждый доступ). Удаление компонента реализовано через приём "swap-and-pop": последний элемент `m_dense` переставляется на место удаляемого, что сохраняет отсутствие пропусков в хранилище за O(1).

Переход на `ComponentPool<T>` также потребовал отказа от `std::type_index` (ключ пулов) и `dynamic_cast<ISavable*>` (проверка поддержки сериализации) — оба места RTTI заменены на `TypeId`, полученный через `getTypeId<T>()`, и на `IComponentPool::getSavable`, резолвящийся на этапе компиляции через `if constexpr (std::is_base_of_v<ISavable, T>)`. Причина - запрет на использование RTTI в coding style Nau Engine (см. [раздел ниже](#соответствие-coding-style-guide-nau-engine)).

**Прошлая реализация лежит в ветке archive/map-based-storage**

Публичный API `Scene` при миграции не менялся (`addComponent`, `getComponent`, `hasComponent`, `getEntitiesWith`, `update`, `saveToFile`, `loadFromFile` те же сигнатуры), поэтому измерения ниже отражают чистый эффект от смены внутреннего хранилища, а ранее написанные `scene_tests`, `movement_tests`, `serialization_tests` и `memory_tests` прошли миграцию без единой правки в самих тестах.

## Профилирование и результаты замеров

Замеры выполнены `scene_benchmark` (встроенный профилировщик `profiler.h/.cpp`, RAII-таймер `ScopedTimer` и макрос `PROFILE_SCOPE`) в конфигурации Release, в сценарии со 100 000 сущностей и 200 вызовами `Scene::update()`.

### До перехода на sparse-set хранилище

| Сущностей | Вызовов update() | Память |
|---|---|---|
| 100 000 | 200 | 17 200 000 байт (16.4032 МБ) |

| Scope | Calls | Total, ms | Avg, ms | Min, ms | Max, ms |
|---|---|---|---|---|---|
| Update loop (total) | 1 | 6307.133 | 6307.133 | 6307.133 | 6307.133 |
| Populate scene | 1 | 85.243 | 85.243 | 85.243 | 85.243 |
| Scene::Update | 200 | 6306.823 | 31.534 | 24.348 | 63.893 |

### После перехода на sparse-set хранилище

| Сущностей | Вызовов update() | Память |
|---|---|---|
| 100 000 | 200 | 8 845 040 байт (8.43529 МБ) |

| Scope | Calls | Total, ms | Avg, ms | Min, ms | Max, ms |
|---|---|---|---|---|---|
| Update loop (total) | 1 | 1328.210 | 1328.210 | 1328.210 | 1328.210 |
| Populate scene | 1 | 13.189 | 13.189 | 13.189 | 13.189 |
| Scene::Update | 200 | 1328.113 | 6.641 | 5.940 | 7.976 |

### Итог

- Потребление памяти снизилось примерно в 1.94 раза (с 16.40 МБ до 8.44 МБ).
- Суммарное время цикла обновления снизилось примерно в 4.75 раза (с 6307 мс до 1328 мс).
- Среднее время одного вызова `Scene::update()` снизилось примерно в 4.75 раза (с 31.53 мс до 6.64 мс).

Прирост объясняется переходом от хранения компонентов по указателю в разрозненных участках кучи к непрерывному хранению по значению внутри `ComponentPool<T>`: итерация по сущностям с заданным компонентом больше не требует косвенных обращений по указателю и лучше использует кэш процессора, а сама структура данных (sparse set) исключает накладные расходы хеш-таблицы на поиск компонента по сущности и по типу.

## Соответствие Coding Style Guide Nau Engine

Код проекта написан с ориентацией на [Nau Engine Coding Style Guide](https://github.com/NauEngine/NauEnginePublic/blob/main/doc/coding_style_guide.md). Стандарт языка выставлен в CMake как C++20.

**Соблюдается:**

- Форматирование: только пробелы, отступ 4 пробела, стиль Allman для фигурных скобок, пробел перед `(` в условных операторах — выдержано по всему коду.
- `#pragma once` вместо include guard — во всех заголовках.
- RTTI запрещено — `typeid`, `dynamic_cast`, `std::type_index` в проекте не используются; вместо них `TypeId`/`getTypeId<T>()` и `IComponentPool::getSavable` с `if constexpr`. Обоснование в `docs/adr/0001-component-pool-storage.md`.
- Касты: только C++-style (`static_cast`), C-style касты не используются.
- `auto` используется точечно (например, для `std::chrono::time_point` в `ScopedTimer`), а не повсеместно вместо явных типов.
- `using` предпочтён `typedef` (`using TypeId = size_t;`).
- Пространства имён: `using namespace` нигде не применяется.
- Документирование: doxygen-стиль (`@brief`, `@param`, `@return`) на всех публичных методах и нетривиальных приватных участках.
- Именование:
  - файлы `snake_case` (`component_pool.h`, `health_component.cpp`);
  - классы `PascalCase` (`Scene`, `ComponentPool`, `HealthComponent`);
  - интерфейсы (только чистые виртуальные функции) префикс `I` (`ISavable`, `IComponentPool`);
  - методы и функции `camelCase` (`addComponent`, `getEntitiesWith`);
  - члены класса префикс `m_` (`m_dense`, `m_sparse`, `m_currentHealth`);
  - статические переменные префикс `s_` (`s_counter` в `type_id.h`, `s_entries` в `profiler.cpp`);
  - константы `UPPER_CASE` и `constexpr` (`INVALID_INDEX` в `ComponentPool`);
  - `&`/`*` часть типа, не имени (`const nlohmann::json& j`, ссылка на `Scene` в `System`, а не указатель, там, где владение не требуется).
- Реализация вынесена в `.cpp` везде, где не мешает шаблонной природе кода; заголовки содержат декларации и тривиальные inline-методы (геттеры вида `current()`, `max()`).
- Композиция предпочтена наследованию там, где это уместно: `Scene` владеет пулами и системами через композицию, а не наследует их; наследование используется только там, где действительно нужен полиморфизм (`Component`/`ISavable` как базовые интерфейсы компонентов, `System` как база систем).
- Const-correctness: геттеры, `toJson()`, `has()`, `entities()`, `memoryBytes()` и другие методы, не изменяющие состояние, помечены `const`.
- Каждая нетривиальная часть покрыта тестами: `scene_tests`, `movement_tests`, `memory_tests`, `serialization_tests`.

**Осознанные отступления** (обоснованы масштабом задания):

- Контейнеры и smart pointers взяты из `std`, а не `eastl`, как того требует guide в общем случае. Задание явно ограничивает проект стандартной библиотекой и разрешёнными сторонними библиотеками уровня `nlohmann/json`/`sol2`; подключение `eastl` ради тестового задания было бы избыточным и не предусмотрено условиями.
- Копирайт-шапка (`// <filename> // Copyright <year> N-GINN LLC...`) не добавлена в файлы.
- `nau::Functor<>` не используется, поскольку является частью экосистемы Nau Engine, не входящей в состав этого изолированного тестового проекта.
